  ## TEMPLATE CONTEXT

  You are planning against an existing Flutter starter template with these locked-in architectural decisions. You MUST NOT propose
   alternatives to these choices — the coding agent will extend this template, not replace it.

  ### Locked-in Tech Stack
  - **Flutter SDK:** `>=3.8.1`
  - **State management:** Riverpod 2.x
  - **Routing:** GoRouter 14.x with ShellRoute wrapping authenticated pages in `ResponsiveScaffold`
  - **Data models:** Freezed (`@freezed`) with `json_serializable` for all immutable models
  - **Theming:** Material 3 with dynamic color seeds (`ColorSeed` enum), Google Fonts (Inter)
  - **Localization:** ARB-based i18n (`flutter gen-l10n`) with English, Spanish, and French
  - **Error handling:** Four-layer system — global (`GlobalErrorHandler`), widget boundary
  (`ErrorBoundary`/`FeatureErrorBoundary`), provider-level (`AsyncValue`), widget-level (`SafeImage`, `CustomAvatar`)
  - **Storage:** `SharedPreferences` for simple key-value persistence, `flutter_secure_storage` available but not yet wired
  - **Backend:** Mock-first via `MockApiService` — all data in-memory with simulated delays; `supabase_flutter` and `dio` declared
   but not wired

  ### Locked-in Project Structure
  lib/
    app.dart
    main.dart
    core/
      constants/          # breakpoints.dart, colors.dart (ColorSeed enum)
      error/              # error_handler.dart (GlobalErrorHandler, ErrorUtils)
      models/             # All @freezed data models
      providers/          # Cross-cutting providers (theme, locale)
      router/             # app_router.dart (GoRouter + redirect logic)
      services/           # mock_api_service.dart
      theme/              # app_theme.dart (Material 3 light/dark)
    features/
      /
        data/             # Repositories (optional, for features with local storage)
        presentation/
          pages/          # Full-screen widgets
          providers/      # Feature-scoped Riverpod providers
          widgets/        # Feature-scoped UI components
    l10n/                 # app_en.arb, app_es.arb, app_fr.arb (+ generated)
    shared/
      widgets/
        error/            # ErrorBoundary, ErrorPage, ErrorWidgets
        file_upload/      # FileUploadWidget
        skeletons/        # SkeletonWidget, page-level skeletons
        ui_components/    # CustomButton, CustomCard, CustomTextField, CustomAvatar, CustomChip
        responsive_scaffold.dart
        safe_image.dart
        search_bar.dart
        guest_mode_banner.dart

  ### Locked-in Provider Patterns
  Use these patterns when defining new providers:
  - **CRUD features:** `StateNotifierProvider<Notifier, AsyncValue<List<Model>>>` — constructor auto-loads, auth-guards mutations,
   holds `Ref` for dependencies
  - **Read-only async data:** `FutureProvider` or `FutureProvider.family`
  - **Simple primitive state:** `StateProvider`
  - **New state-logic providers:** `@riverpod` code generation
  - **DI for services:** Plain `Provider`
  - **Computed values:** Derived `Provider` that `.watch()` other providers

  ### Locked-in Routing Conventions
  - Standalone routes (no scaffold): `/login`, `/register`, `/onboarding`
  - Authenticated routes inside `ShellRoute` with `ResponsiveScaffold`: `/home`, `/dashboard`, etc.
  - Detail pages as nested sub-routes: `/users/:userId`
  - Redirect logic in `routerProvider` handles onboarding completion and auth state
  - Navigation destinations configured in `ResponsiveScaffold` (bottom nav <1000px, compact rail 1000–1500px, extended rail
  >1500px)

  ### Locked-in Shared Widget Library
  Always prefer these over building new widgets:
  - `CustomButton` — 4 variants (primary/secondary/tertiary/danger), 3 sizes, loading state
  - `CustomCard` — 3 variants (elevated/filled/outlined), tap handler, animation
  - `CustomTextField` — outlined/filled variants; `EmailTextField` and `PasswordTextField` presets
  - `CustomAvatar` — 4 sizes, circle/rounded, image/initials/icon fallback, `AvatarGroup`
  - `CustomChip` — 4 variants, filter/selection state, `ChipGroup`
  - `AppSearchBar` — search icon + clear button
  - `ErrorBoundary` / `FeatureErrorBoundary` — widget-level error catching
  - `SkeletonWidget`, `SkeletonText`, `SkeletonCard`, etc. — loading states
  - `FileUploadWidget` — image/document/any with preview
  - `GuestModeBanner` — auth prompt for unauthenticated users
  - `SafeImage` — network image with error fallback

  ### Locked-in Model Conventions
  - All models in `lib/core/models/` with `@freezed` and `fromJson`/`toJson`
  - Enums defined alongside their model in the same file
  - Nested models (e.g. `DashboardStats` containing `ChartData`, `ActivityItem`) in one file
  - Generated part files: `*.freezed.dart` and `*.g.dart`

  ### Locked-in Localization
  - All user-visible strings must use `AppLocalizations.of(context)!.keyName`
  - Add keys to all three ARB files: `app_en.arb` (source with `@key` metadata), `app_es.arb`, `app_fr.arb`
  - Parameterized strings: `"greeting": "Hello, {name} {count, plural, =1{item} other{items}}"`
  - After editing ARB files, run `flutter gen-l10n`

  ## OUTPUT

  Produce a structured implementation plan in the following format. Be specific, technical, and code-generation-ready — every
  decision will be consumed directly by an AI coding agent working within the starter template.

  ### 1. App Overview
  - App name, purpose, target platform(s)
  - Target users and core use case in 2–3 sentences

  ### 2. Template Modifications
  List what needs to change in the starter template's existing files. For each:
  - File path (relative to project root)
  - What to modify and why
  - Be explicit: new routes to register, new navigation destinations, new ARB keys, new models to add, providers to register, etc.

  ### 3. New Feature Modules
  For each new feature under `lib/features/<name>/`:
  - Feature name
  - Directory structure (pages/, providers/, widgets/, data/)
  - Detailed description

  ### 4. Screen Inventory
  List every screen in the finished app. For each:
  - Screen name and Widget class name (e.g. `TaskListPage`)
  - Route path (GoRouter format, e.g. `/tasks`)
  - Whether it sits inside the ShellRoute (authenticated) or is standalone
  - Purpose (1 sentence)
  - Which shared widgets it uses
  - Which providers it reads/watches
  - User actions available
  - Localization keys needed

  ### 5. Feature Breakdown
  For each feature derived from the user's answers:
  - Feature name
  - Detailed description
  - Acceptance criteria (bullet list)
  - Dependencies (other features, device permissions, existing template components)
  - Priority: Must-have / Should-have / Nice-to-have

  ### 6. Data Models
  Define all new Dart data classes that need to be added to `lib/core/models/`. For each:
  - Class name
  - Fields with Dart types, nullability, and `@Default` values
  - Relationships to other models (including existing template models like `User`)
  - Confirm: uses `@freezed` + `json_serializable` with generated part files

  ### 7. Provider Definitions
  For each new Riverpod provider:
  - Provider name and type (`StateNotifierProvider`, `FutureProvider`, `StateProvider`, `@riverpod`, or plain `Provider`)
  - State type (e.g. `AsyncValue<List<Task>>`)
  - Which other providers it depends on (`ref.watch`, `ref.read`)
  - Methods (for StateNotifier: load, create, update, delete, etc.)
  - Auth-gating behavior (does it check `authProvider` before mutations?)

  ### 8. Local Storage Design (if applicable)
  For any data that must persist on-device:
  - Entity name and storage mechanism (SharedPreferences keys, or a new service if needed)
  - Fields persisted with types
  - Read/write patterns
  - Whether this extends `MockApiService` or requires a new persistence layer

  ### 9. Routing Changes
  Exact changes to `lib/core/router/app_router.dart`:
  - New routes to add (with path, widget, and whether inside or outside ShellRoute)
  - New sub-routes (e.g. `/tasks/:taskId`)
  - Any new redirect logic
  - New navigation destinations to add to `ResponsiveScaffold`

  ### 10. Localization Keys
  All new ARB keys grouped by screen/feature. For each key:
  - Key name
  - English value (with parameter placeholders if applicable)
  - Context (which screen uses it)

  ### 11. Implementation Phases
  Break the build into sequential phases the AI coding agent can execute one at a time:
  - Phase number and name
  - Goal of the phase
  - Ordered list of tasks (granular enough to be individual coding steps)
  - Specific files to create or modify (use exact paths)
  - Which `build_runner` or `gen-l10n` steps are needed after the phase
  - Definition of done for the phase

  ### 12. Assumptions & Open Questions
  - List every assumption you made due to missing or ambiguous input
  - Flag any decisions the user must confirm before code generation begins
  - No backend implementation as for now, keep data local

  ## RULES

  - All code must be valid, idiomatic Dart/Flutter that extends the existing starter template.
  - Do not propose changes to the locked-in tech stack or architectural patterns — work within them.
  - Do not invent features not implied by the user's answers.
  - Every class name, route, widget, provider, and ARB key must be consistent across all sections.
  - Use precise technical language — this plan will be read by an AI coding agent, not a stakeholder.

  - All new data models go in `lib/core/models/` and use `@freezed` + `json_serializable`.
  - All user-facing strings must go through `AppLocalizations` with keys in all three ARB files.
  - If the user's answers conflict or are contradictory, surface the conflict clearly and propose a resolution.
  - Keep the plan self-contained: the coding agent will receive only this document, so nothing can be left implicit.
  - If the app is 100% local with no backend, extend `MockApiService` with the new entities and methods. If a backend is needed,
  note that `supabase_flutter` and `dio` are available but not yet wired.
